better inlining — idea list (from the builtins-in-aipl work; none implemented yet):
- Inline specialized HOF instances at their call sites: a `__builtin_all$i64$lamN`
  (or `count_while$...$lamN`) instance typically has exactly one call site, so
  inlining it and not emitting the function is a pure win (call overhead + binary
  size). The spec_memo already knows when an instance is reused.
- Fuse the lifted lambda into the specialized HOF body: after
  `specialize_call_with`, the loop calls `__lambda_N($e, caps..)` through a
  statically-known binding — inlining the predicate into the loop removes a call
  per element. Probably the single highest-value inline for AIPL-implemented
  builtins like `all`.
- Pick the layer: AST-level in mono (cheap — mono already synthesizes/rewrites
  bodies; but must lower early `return` in an inlined body, e.g. `all`'s
  `return false`, into a value-producing form) vs Cranelift-IR-level (handles
  every body shape, bigger lift; check what cranelift's own inlining infra
  offers now).
- Heuristics: always inline single-call-site instances; otherwise a body-size
  threshold; never inline recursive fns; memoized instances with many call
  sites stay outlined to avoid bloat.
- Synergy: inlining exposes retain/release pairs across the old call boundary —
  refcount elision after inlining likely beats the call-overhead win itself.

Lex aipl:
  New matcher shapes (fit the current data-only design)

  3. Escape-aware delimiter scanning — in "…" a \" must not close the string (lib.rs:2404). The close-scan needs to skip \<any> pairs.
  4. Char-literal shape — 'x' / '\n' (lib.rs:2469): open quote, optional escape, exactly one char, close quote. A same-char Delimited("'") gets simple cases right but '\'' wrong, and can't reject '' (empty) or multi-char.

  New lexer machinery (changes to lex itself)

  6. Trivia side-channel. Skip rules currently drop spans, but the real lexer records every comment span (record_comment, lib.rs:2265 — the formatter depends on this) and every #[allow] span (record_allow, lib.rs:2283 — the lint pass depends on it). lex needs to return skipped/marker spans alongside the tokens (e.g. a result struct { tokens, comments, allows } — needs struct-with-array-fields as a result payload, or a second out-array).
  7. Template literals — mode-switching re-entrant lexing. The big one. `text {expr} text` (lib.rs:2719) alternates verbatim-text scanning (with its own escape set \\ \ { \n \t \r) and full-rule-set expression lexing inside {…} with brace-depth tracking (tokenize_template_expr, lib.rs:2910), emitting *context-dependent* kinds (STR vs HEAD/MIDDLE/TAIL). Triple-backtick ```` ``` ```` (lib.rs:2964`) adds verbatim (no-escape) text with the raw-string dedent treatment. This doesn't fit "one matcher, one token" at all — it needs sub-lexing or a lexer-mode stack.
  8. Unicode identifiers and spans. The Rust lexer is char-based: idents use is_alphabetic()/is_alphanumeric() (full Unicode, so héllo lexes today), and error spans use len_utf8. CharRun's char-class strings are byte sets — they can't express "any Unicode letter", and a multi-byte char would be split. Either add a named-class matcher (e.g. Alphabetic) or decide AIPL idents become ASCII-only (a language decision, not a lexer one).

  Deferred by design (value processing, not tokenization)

  9. Token values. Terminal carries processed payloads — escaped string contents, the raw-string dedent (via the already-dogfooded process_raw_string), char byte values, parsed i64 (with overflow reported at lex time, lib.rs:2597). Our tokens are span-only; slicing covers idents/numbers, but string unescaping and the overflow check need a post-pass per kind. process_raw_string can be reused as-is.

  Rough dependency order if you want a sequence: 5 (error machinery) unblocks 1–4 (the new matchers all need hard errors); 6 is independent and small; 7 is the largest and worth its own design pass; 8 needs your call on ASCII vs Unicode first.
